# CAA 知识库蒸馏规则

> 版本: 1.0 | 更新: 2026-05-13

---

## 一、核心原则

### 1.1 知识分层架构

知识库采用**洋葱模型**三层结构，从外到内依次为：

```
Layer 3  深度知识     knowledge/tier*/  概念理解、架构导航、设计模式
Layer 2  结构化数据   data/*.json        精确查询、方法索引、交叉引用
Layer 1  原始文档     api-ref/, use-cases/ 方法签名、完整代码、用例细节
```

**每一层的职责边界清晰，不可混淆：**

| | Layer 1 | Layer 2 | Layer 3 |
|---|---|---|---|
| 怎么产生 | 脚本自动生成/转换 | 脚本提取+索引 | 人工阅读后内化输出 |
| 给 agent 提供什么 | 精确答案 | 快速定位 | 理解框架 |
| 更新频率 | 源码变化时 | 源码变化时 | 阅读推进时 |
| 可否独立使用 | 需要 Layer 2 导航 | 需要 Layer 1/3 补充 | 需要 Layer 1/2 精确化 |

### 1.2 不可用脚本替代人工理解

- 脚本负责**结构化提取**（方法签名、索引、交叉引用）——可批量、可重复
- 人工负责**知识内化**（架构理解、设计意图、模式归纳）——不可批量
- 脚本产出的内容不称为「蒸馏知识」，称为「结构化数据」
- 只有经过人工阅读、理解、归纳后输出的文档才称为「蒸馏知识」

### 1.3 一切以 AI agent 可用性为目标

每份产出在提交前必须回答三个问题：
1. agent 在什么场景下会用到这份内容？
2. agent 如何找到这份内容？
3. agent 拿到这份内容后能否直接使用（无需再读源码）？

## 二、Layer 3 深度知识文档规范

### 2.1 目录结构

```
knowledge/
├── tier1/           # 核心基础框架（必须先读）
│   ├── CAASystem-核心框架.md
│   ├── CAAApplicationFrame-应用框架.md
│   └── ...
├── tier2/           # UI/对话框
├── tier3/           # 机械建模核心
├── tier4/           # 产品结构
├── tier5/           # 加工制造
├── tier6/           # 专业领域
└── tier7/           # 小型/工具项目
```

Tier 分级依据：
- **Tier 1**: 所有其他模块的依赖基础，定义 CAA 核心范式
- **Tier 2**: UI 框架，几乎所有应用都需要的交互层
- **Tier 3-5**: 按业务领域划分，有明确的依赖方向
- **Tier 6-7**: 独立性强的小型模块

### 2.2 文档模板

每个模块的蒸馏文档必须包含以下章节：

```markdown
---
module: "{.edu 项目名}"
category: "{一句话定位}"
tier: "{1-7}"
status: "{已完成|进行中}"
---

# {项目名} — {中文标题}

## 模块定位
{2-3 句话说明这个模块在 CAA 体系中的位置和作用}

## 架构总览
{ASCII 图或文字描述模块内部结构和数据流}

## 核心接口详解
{逐个讲解最重要的 PublicInterface，包含:}
- IID 和基类
- 每个方法的签名和用途（不是复制粘贴，是解释意图）
- 与其他接口的协作关系

## 实现模式
{本模块特有的实现技巧，含代码示例}

## 关键设计模式
{本模块体现的 CAA 设计模式}

## 对 AI agent 的要点
{列出 agent 帮助开发者时可以直接引用的知识点}
```

### 2.3 写作标准

- **方法说明要解释「为什么」而非「是什么」**：
  - ❌ `SetContainer(CATBaseUnknown*)` — 设置容器
  - ✅ `SetContainer` 将组件关联到文档容器，使 QueryInterface 能在组件上查找容器实现的接口——这是 CAA 最核心的组件-容器关联模式

- **代码示例要完整可运行**：不能只写片段，必须包含 include、QueryInterface、错误检查

- **必须标注信息来源**：每个结论要能从源码中找到出处

- **中文描述 + 英文保留术语**：接口名、宏名、类型名保持英文原样

### 2.4 与 Layer 1/2 的链接

每个 Layer 3 文档必须包含指向 Layer 1/2 的链接：

```markdown
## 相关资源
- 完整方法签名: [api-reference/interfaces/](api-reference/interfaces/)
- 结构化查询: [knowledge_base.json → method_index](data/knowledge_base.json)
- 使用案例: [cross_references.json](data/cross_references.json)
```

反之，Layer 1 的接口文档末尾也应添加指向 Layer 3 的链接：
```markdown
## 深入理解
参见 [{模块名} 核心框架](knowledge/tier{N}/{模块名}-{标题}.md)
```

## 三、Layer 2 结构化数据规范

### 3.1 数据文件清单

| 文件 | 用途 | 生成方式 |
|------|------|----------|
| `knowledge_base.json` | 接口方法签名 + method_index | 脚本: extract_interface_methods.py |
| `search-index.json` | 关键词 → 文档路径 | 脚本: rebuild_index.py |
| `cross_references.json` | 接口 ↔ use-case 双向链接 | 脚本: rebuild_index.py |
| `interface_methods.json` | 完整的接口方法提取结果 | 脚本: extract_interface_methods.py |
| `knowledge-index.json` | 统一入口 + 查询指南 | 脚本: finalize.py |

### 3.2 更新规则

- 源码变化时重新运行对应脚本
- 新增 Layer 3 文档时，更新 `search-index.json` 将其纳入索引
- 新增 Layer 3 文档时，更新 `knowledge-index.json` 的 stats 和 query_guides

### 3.3 不可包含的内容

- ❌ 报告文件（*报告*.md、*执行*.md、*审查*.md）
- ❌ 计划文件（*计划*.md、*TODO*.md）
- ❌ 重复的旧版本数据
- ✅ 只包含技术文档和代码索引

## 四、Layer 1 原始文档规范

### 4.1 接口文档 (api-reference/interfaces/)

每个接口一个 .md 文件，包含：
- YAML frontmatter（title, type, module, base, method_count, source_file）
- 基类、模块、方法数摘要
- 依赖（include 列表）
- 方法签名（按纯虚/虚/静态/公共分组）
- 每个方法的参数表

**质量标准**:
- Frontmatter 正确率: 100%
- 方法签名可查询率: ≥ 79%（受限于源码本身的空接口）
- 源文件路径: 必须指向实际 .h 文件

### 4.2 Use-Case 文档 (use-cases/)

从 HTML 转换而来的使用案例，每个 .md 包含：
- YAML frontmatter（title, category, module, tags, source_file）
- 原始文档的文字内容和代码示例

**质量标准**:
- Frontmatter 正确率: 100%
- 代码围栏配对: 100%
- 无重复段落

## 五、阅读执行流程

### 5.1 单个模块的阅读步骤

```
第1步: 列出 PublicInterfaces/*.h       → 理解对外契约
第2步: 逐个阅读每个 .h                  → 记录方法、参数、注释
第3步: 列出 ProtectedInterfaces/*.h    → 理解扩展点
第4步: 阅读 LocalInterfaces/*.h        → 理解实现类
第5步: 阅读关键 .cpp                   → 理解 TIE、CATImplementClass 等模式
第6步: 归纳架构                         → 画数据流图
第7步: 提取设计模式                     → 对照 CAA 标准模式
第8步: 输出 Layer 3 文档                → 按模板填写
第9步: 更新 Layer 2 索引                → 将文档纳入搜索
```

### 5.2 阅读顺序

严格按 Tier 1 → 7 的顺序阅读，不跳级。原因：
- Tier 1 定义的基础概念（接口、组件、扩展、QueryInterface）是后续所有模块的前提
- 高层模块的代码引用了低层模块的接口，不先读低层则无法理解

### 5.3 质量检查

每完成一个模块的文档，自检：
- [ ] 是否覆盖了所有 PublicInterface？
- [ ] 每个方法的说明是否解释了「为什么」？
- [ ] 是否包含可运行的代码示例？
- [ ] 是否标注了信息来源？
- [ ] 是否链接了 Layer 1/2 的相关资源？
- [ ] agent 能否仅凭此文档理解模块用途？

## 六、禁止事项

- ❌ 不得用脚本批量生成 Layer 3 文档
- ❌ 不得跳过阅读直接复制方法签名到 Layer 3
- ❌ 不得在 Layer 3 中重复 Layer 1 的完整方法签名表
- ❌ 不得在 Layer 2 索引中纳入报告/计划文件
- ❌ 不得在 Layer 1 文档中写入未经源码验证的代码示例
- ❌ 不得删除旧版数据文件前不确认其引用关系
