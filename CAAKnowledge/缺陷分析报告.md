## 🔴 高优先级缺陷

### 缺陷 1: Use-Case 文档路径损坏（破损的相对引用）

**严重性**: 高  
**影响范围**: ~100 个 Use-Case 文档  
**现象**: HTML 转换后的 Markdown 文档中残留大量 `../` 相对路径引用，指向原始 HTML 文件位置，无法在 Markdown 文件树中正常工作。

**示例**:
```markdown
| ![image](../../assets/images/atarget.gif) | This macro shows... |
![image](../../assets/images/aendtask.gif)
[Top]
[Creating Connection Properties on a Product](../use-cases/caascdaniusecases/...)
```

**根因**: V1/V2 HTML 转换器在提取 HTML 表格内容时，直接复制了原始的 `../` 相对路径，未做路径重映射。

**影响**:
- 图片无法正常显示（路径指向 `Doc/online/CAAScrBase/images/` 而非知识库目录）
- 内部链接指向 `.htm` 文件而非 `.md` 文件
- 文档间的交叉引用全部失效

**建议修复**:
1. 重写 V2 转换器的路径处理逻辑：
   - 检测 `../` 开头且指向 `images/` 的图片路径 → 映射到知识库对应模块的 `images/` 目录
   - 检测指向 `.htm` 的链接 → 转换为指向对应的 `.md` 文件（去除 `.htm` 后缀）
   - 保留指向外部资源的绝对 URL 不变
2. 运行路径修复脚本扫描所有文档

### 缺陷 2: HTML 表格残留（未转换的表格格式）

**严重性**: 高  
**影响范围**: ~1149 个文档  
**现象**: HTML `<table>` 转换为 Markdown 后生成大量单列两行 `|` 包围的表格，而非标准多列 Markdown 表格。

**示例** (当前输出):
```markdown
| 

## Creating Connection Properties on a Product  
  
---|---  
```

**问题分析**:
- HTML 源文件使用两列布局的表格：一列是图标列，一列是内容列
- 当前转换器将 `<tr><td>...</td><td>...</td></tr>` 全部合并成单列 `|` 表格
- 正确的做法应该是检测到 `<td class="use">` 或图标列后跳过，仅保留内容列

**建议修复**:
1. 在 V3 转换器中增加 `<table>` 结构的特殊处理：
   - 检测 `<td class="use">` / `<td class="cen">` / 图标列 → 跳过该列
   - 检测首行含 `|---` 样式的表头行 → 提取剩余列内容
   - 合并跨行单元格时正确展开
2. 补充测试：验证至少 5 个不同模块的表格转换结果

### 缺陷 3: category 字段全为 "general"（分类元数据缺失）

**严重性**: 高  
**影响范围**: 100% Use-Case 文档  
**现象**: 所有文档的 frontmatter 中 `category` 字段均为 `"general"`，未反映实际内容分类。

**当前 frontmatter**:

**根因**: V2 转换器的 `_extract_frontmatter(#)` 方法中，category 提取逻辑存在但仅在检测到 `class="use"` 的 `<td>` 时才有效。由于 HTML 表格结构复杂，该检测经常失败。

**建议修复**:
1. 从文档标题推断：`Creating` → use-case、`How to` → tutorial、`Overview` → concept
2. 从模块目录名推断：`CAAScdAniUseCases` → `use-cases`、`CAAAScdAniTechArticles` → `tech-article`
3. 从源码注释推断：查找 `Scenario:`、`Example:`、`Sample:` 等关键词

### 缺陷 4: 479 个空接口文档（信息量不足）

**严重性**: 高  
**影响范围**: 772 个接口文档中的 479 个（62%）  
**现象**: 这些接口没有声明任何方法，仅有基类和少量元数据。

**问题分析**:
```markdown
## 说明

该接口没有声明自定义方法，作为标记接口或配置接口使用。
```

**实际情况**: 很多"空接口"并非真的无意义，可能是：
1. 方法在基类或祖先类中声明
2. 方法是委托给其他组件实现的
3. 接口仅用于类型标记（marker interface）

**建议修复**:
1. 对 479 个空接口进行继承链追溯：
   - 提取基类名
   - 查找基类的 `.h` 文件
   - 如果基类有方法，则在文档中注明"方法继承自 [基类名]"
2. 添加接口用途说明：从 `.cpp` 文件的 TIE 绑定注释中提取接口使用场景
3. 标记为 marker interface 并说明典型用途
