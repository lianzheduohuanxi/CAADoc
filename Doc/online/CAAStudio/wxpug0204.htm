Creating a Module

This task shows you how to create a module in a framework.
Before you begin:

Open a workspace containing at least a framework.

  1. In the 3DS Workspace Explorer, right click the name of framework for which you want to create a module, and click New Module.

    The 3DS - New Module dialog box opens.

  2. In Name, type the name you want to assign to the module. Refer to the CAA V5 C++ Naming Rules.

  3. In the Module type list, select the appropriate module build type.

    The module type is declared in the Imakefile.mk file.

    Type Description Imakefile.mk
    Dynamic Link Library (dll) Windows Dynamic-Link Library (.dll). BUILT_OBJECT_TYPE=SHARED LIBRARY
    Executable (exe) Windows executable file (.exe). BUILT_OBJECT_TYPE=LOAD MODULE
    Archive (lib) Windows library (.lib). BUILT_OBJECT_TYPE=ARCHIVE
    None Module not built. BUILD=NO
    Web module (.mweb) Web module (.mweb). BUILT_OBJECT_TYPE=WEB
    External web module (.mwebext) External web module (.mwebext). BUILT_OBJECT_TYPE=WEB
    Resource Run time resource module (.m). BUILT_OBJECT_TYPE = RUNTIME DATA

  4. In Header exposition, select the scope of the module's APIs.

    This sets the location of the module ExportedBy file and must match the higher level of APIs this module exposes.

    Scope Description
    Private: header only visible for this framework The module's APIs scope is private. The ExportedBy file and the API header files are located in the PrivateInterfaces folder.
    The common usage is that the module's APIs are framework internal APIs that can be called from the framework only. The private scope must usually not be used to access these internal APIs from other frameworks.
    Protected: header shared with team The module's APIs scope is protected. The ExportedBy file is located in the ProtectedInterfaces folder.
    The common usage is that some or all of the module's APIs are modeler of component internal APIs that can be called from the framework and from the frameworks that collaborate to make up the modeler or the component, thus "header shared with the team". These API header files are located in the ProtectedInterfaces folder. The protected scope must usually not be used to access these APIs from other frameworks that do not collaborate to make up to the modeler or component.
    Nevertheless, some other module's API headers can be located in the PrivateInterfaces folder and are not part of the protected scope, but of the private scope.
    Public: header with an external visibility The module's APIs scope is public. The ExportedBy file is located in the PublicInterfaces folder.
    The common usage is that some or all of the module's APIs are external APIs that can be called from any frameworks. These API header files are located in the PublicInterfaces folder.
    Nevertheless, some other module's API headers can be located in the ProtectedInterfaces and/or PrivateInterfaces folders and are not part of the public scope, but of the protected or private scope.

  5. Click Finish.

    The module is created.
